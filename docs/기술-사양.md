# 헐버트작가촌 사이트 — 기술 사양서

> 🏷️ **📝 초안** · 갱신 2026-07 · 사이트 기술 사양 정본

> [`홈페이지-사이트맵`](홈페이지-사이트맵.md)(구조 정본)과 [`pages/`](pages/) 카피(내용 정본)를 **실제 공개 사이트로 만드는 방법**을 정의합니다.
> 개발자(또는 Claude Code 세션)가 이 문서 하나로 착수할 수 있게 — 스택 · 저장소 구조 · 빌드/배포 · 폼 · SEO · 성능 기준을 확정합니다.

**버전 v0.1** &nbsp;|&nbsp; ⚠️ `결정 필요` 표시 항목은 운영자 결정 대기. 나머지는 이 문서가 기준.

## 0. 원칙

1. **저장소에서 검증된 방식을 재사용한다** — 새 프레임워크 없이, 문서 뷰어에서 이미 작동 중인 `Python(markdown) 정적 빌드 → GitHub Pages 자동 배포`를 그대로 확장한다. 예산 0원, 1인 유지보수 가능.
2. **문서가 원천, 사이트는 렌더링** — 카피 수정은 언제나 `docs/`의 .md에서 하고, 사이트는 빌드 산출물이다. 손으로 HTML을 고치지 않는다.
3. **예시는 예시로 보이게** — 미확정 수치(수수료 구간·로테이션 4주 등)는 렌더링 후에도 '예시' 라벨이 보존되어야 한다(디자인-가이드의 '예시 배지' 컴포넌트 사용). 이는 [`시장-벤치마크-개관-전략`](시장-벤치마크-개관-전략.md) §7과 브랜드 원칙(정직)의 기술적 구현이다.

## 1. 스택

| 항목 | 결정 | 근거 |
| --- | --- | --- |
| 사이트 생성 | **Python 정적 빌드** (`site/build_site.py` 신설, `markdown` 라이브러리) | 뷰어(`viewer/build.py`)로 검증됨 · 의존성 1개 · Claude Code 세션으로 전 과정 수행 가능 |
| 템플릿 | Python 문자열 템플릿 + 페이지별 HTML 골격 | 뷰어와 동일 방식 · 빌드 도구 추가 없음 |
| 스타일 | 순수 CSS 1파일 (`site/assets/style.css`) — [`디자인-가이드`](디자인-가이드.md) 토큰 사용 | 프레임워크 불필요 규모(9페이지) |
| JS | 최소한(아코디언·모바일 메뉴 정도), 없어도 읽히게 | 뷰어 성능 원칙 계승 |
| 호스팅 | **GitHub Pages** (현행 유지) | 무료 · `main` 푸시 시 자동 배포 이미 구축됨 |
| 도메인 | Pages 기본 URL로 시작 → 커스텀 도메인 확보 시 CNAME 연결 | 도메인 선점은 로드맵 Phase 1-1 |

> `결정 필요` — Vercel(vercel.json) 병행 유지 여부: Pages 단일화 권장(관리 지점 1개). 유지 시 rewrite 목록을 사이트 구조에 맞게 갱신해야 함.

## 2. 저장소 구조 (목표)

```
docs/                  ← 원천 문서 (현행 유지)
site/
  build_site.py        ← 사이트 빌더 (신설)
  config.json          ← 실데이터 단일 변수 파일 (신설, §4)
  templates/           ← 페이지 골격 (신설)
  assets/style.css     ← 디자인 토큰 기반 스타일 (신설)
  assets/img/          ← 사진·일러스트
_site/ (빌드 산출물)    ← Pages 아티팩트 루트
  index.html …         ← 사이트 = 루트
  viewer/              ← 문서 뷰어 = /viewer/ 하위 (내부 도구로 격하)
```

- 사이트가 **공개 정본**, 뷰어는 `/viewer/` 하위의 **내부 문서 도구**가 된다. README·문서-지도의 안내 링크도 그에 맞게 갱신한다.
- 뷰어 격하 시점에 **미검토 법률 초안 노출 통제**(로드맵 Phase 0-2)를 함께 처리한다.

## 6. SEO · 메타

- 페이지별 `<title>`·`<meta description>` — 카피 문서의 제목·요약에서 생성.
- **OG 태그**(제목·설명·대표 이미지) 전 페이지 — SNS 공유가 모집 채널이므로 필수.
- `sitemap.xml`·`robots.txt` 빌더가 자동 생성.
- `결정 필요` — 뷰어(`/viewer/`)의 `noindex` 처리 여부: 내부 도구 격하와 함께 **noindex 권장**(미검토 초안이 검색에 잡히지 않게).
- 언어: `lang="ko"` 단일 (사이트맵 정본 — 언어 토글 없음).

## 7. 성능 · 접근성 기준 (뷰어 최적화에서 계승)

- 초기 페이지 전송(gzip) **100KB 이하** 목표, 첫 화면 히어로만 우선 로드(`fetchpriority`) — 나머지 이미지 `loading="lazy"`.
- 웹폰트(Pretendard) preconnect + 비차단 로드, 시스템 폰트 폴백.
- JS 없이도 전체 내용이 읽히는 구조(정적 HTML 우선).
- 시맨틱 마크업(`header/nav/main/footer`, 제목 위계) · 텍스트 대비 WCAG AA 이상(디자인-가이드 토큰에서 보장) · 키보드 탐색 가능한 아코디언/메뉴.
- 검증: 배포 전 Lighthouse(또는 Playwright 스모크) — 뷰어에서 쓰던 테스트 방식 재사용.

## 8. 빌드 · 배포 파이프라인

```
docs/*.md + site/config.json
  → python3 site/build_site.py   (사이트 렌더)
  → python3 viewer/build.py      (뷰어 렌더 — 현행)
  → GitHub Actions(deploy-pages.yml): 위 2개를 CI에서 실행 후 산출물 업로드
```

- `deploy-pages.yml`에 **빌드 단계 추가**(setup-python → `pip install markdown` → 빌드 2종) — 커밋만 하면 항상 최신이 배포되게. *(로드맵 Phase 0-3 — 수동 재실행 의존 제거)*
- `gen-images.yml`의 커밋 대상에 `viewer/sources.json` 누락 수정(같은 커밋에서).
- 롤백: Pages는 커밋 단위 배포이므로 `git revert` → 푸시로 즉시 복구.
